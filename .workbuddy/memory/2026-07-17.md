## 设计文档：保证金（押金 / Escrow）模块
- 产出：`docs/deposit-module-design.md`（中文，镜像 `trading-mode-strategy-design.md` 的 12 节结构）。
- 定位：与 auction 模块**解耦**的新模块 `internal/modules/deposit/`（六边形 + CQRS + Fx）。DB 的 `deposits` 表是「意图账本/状态机」，真实锁资/释放/扣款经 `PaymentPort` 委托外部 PSP；v1 用 `MockPaymentAdapter`（永远成功）跑通全链路，不落地真实资金。
- 关键设计：押金粒度 `(user_id, auction_id)`；`DepositStatusEnum` 五态 `pending/held/released/applied/forfeited`；资格审查走端口 `DepositGuard`（deposit 实现、auction 注入），在 `PlaceBid` 同步预校验 + `BidProcessor` 异步强校验（`ErrDepositNotHeld` 入 `IsPermanentBidError` → DLQ）；结算走事件——deposit 订阅 `AuctionEndedEvent` → 释放非中标者 / 应用中标者，auction 不反向依赖 deposit。
- 迁移：`000009_create_deposits_table.sql` + `000010_add_auction_deposit_config.sql`（单文件 `-- +goose Up/Down`；金额 `bigint` 分；状态 `varchar` + `CHECK`；`UNIQUE(user_id, auction_id)`）。事件 `deposit.evt.<userID>` → 新增 `DEPOSIT_EVENTS` 流；建议把 outbox relay/envelope 下沉为 `internal/shared/modules/outbox` 共享。
- 勘察确认的事实（避免再错）：迁移实为单文件 goose 标记（非 `.up`/`.down` 拆分，之前 git status 瞥见的是陈旧视图）；事件链路为「domain event → envelope → `event_outbox` 表 → relay → JetStream → consumer → websocket hub」（无 dispatcher 文件）；`event_outbox` 是共享表；用户身份即 `uint64 UserID`（bidder 是 authn 角色）。

  - `internal/modules/deposit/infra/payment/factory.go`：`NewPaymentPort(logger)` 读 `config.GetConfig().Payment.Provider` 在 `MockPaymentAdapter`(默认/"mock") 与 `GenericPaymentAdapter`("generic") 间切换，未知值回退 mock 并 `Warn`。`module.go` 将 `payment.NewMockPaymentAdapter` 改为 `payment.NewPaymentPort`。
  - 测试 `generic_payment_adapter_test.go`（httptest 验证四动作请求构造/回退/错误码 + 工厂默认走 mock）。注意 `logger.New(config.Config{})` 会因 `MustLogLevel("")` panic，测试须传 `config.Config{Log:{LogLevel:"info"}}`。

## listing 模块：多级分类（materialized-path 树）
- 用户指令「listing模块中新增多级分类」。原本已是邻接表 `parent_id`（仅单层 `?parent_id=` 列表），缺 depth/path、最大层级约束、子树查询与嵌套树 API。
- 落地：
  - 迁移 `000011_add_category_hierarchy.sql`：`categories` 加 `depth INT NOT NULL DEFAULT 0`、`path TEXT NOT NULL DEFAULT ''`，建 `idx_categories_path`；递归 CTE 回填既有数据（root depth=0，path=`/<id>`）。
  - `infra/query/category.sql` 新增 `FinalizeCategoryHierarchy`(INSERT 后据父级算 depth/path 并 RETURNING)、`ListAllCategories`、`ListCategoryDescendants`(path 前缀 `LIKE parent.path || '/%'`)。改完跑 `sqlc generate`（v1.30.0）。
  - domain `category_model.go`：加 `depth`/`path` 字段 + `Depth()`/`Path()`；`RestoreCategoryModel` 签名增加 `depth int32, path string` 两个参数（所有调用点/测试已同步）；新增 `MaxCategoryDepth = 6`（根 0 → 最深 7 层）。`errs` 新增 `ErrCategoryDepthExceeded`。
  - `CreateCategoryCommand`：父存在性校验后加 `if parent.Depth()+1 > MaxCategoryDepth → ErrCategoryDepthExceeded`；output 带 Depth/Path。
  - ports `CategoryRepository` 加 `ListAll`/`ListDescendants`；`infra/repository/category_repository.go` 实现二者，`Create` 改为先 INSERT 再 `FinalizeCategoryHierarchy`。
  - 新查询 `GetCategoryTreeQuery`：RootID=nil 返回全森林（ListAll 内存建树），否则 FindByID+ListDescendants 返回以 RootID 为根的子树；`buildCategoryForest` 按 parentID 链接组装嵌套 `Children`。
  - DTO/Handler/Router：`CategoryResponse` 加 Depth/Path；新增 `CategoryTreeNode`+`CategoryTreeResponse`；`CategoryHandler.Tree`/`Subtree`；路由 `/tree`、`/{id}/tree`。`module.go` 注入 `NewGetCategoryTreeQuery`。
  - 测试：regenerate mocks (`mockery` `all:true`)；新增 model Depth/Path 测试、create 命令深度超限测试、tree 查询森林/子树/根不存在测试。
- 验证：全绿。`go build/vet/test ./...` 通过；`golangci-lint run ./internal/modules/listing/... ./tests/...` 0 issues（nestif 曾告警，已把 `Execute` 拆出 `loadSubtree` 提前返回）；`nilaway --include-pkgs=auction ./internal/modules/listing/...` exit 0。README 新增「Category Endpoints (multi-level)」小节。
- 关键坑：`CreateCategory` SQL 不 INSERT depth/path（由 Finalize 计算），故 `CreateCategoryParams` 无这俩字段——`ToCreateParams` 不能塞 Depth/Path，否则 sqlc 报 unknown field 且 mockery 解析失败。
