# PRD — 以 NATS.io 作为竞拍核心与消息通知（移除 Redis）

## Overview

在线竞拍系统当前用 Redis Pub/Sub 作为实时通知总线：出价、竞拍开始/结束等事件经 Redis 频道扇出到 WebSocket 客户端。Redis Pub/Sub 是**即发即弃、无持久化**的——节点重启或订阅者短暂离线都会丢失事件，也无法回放或审计出价历史。同时出价处理为同步落库，缺少削峰与解耦能力。

本方案用 **NATS.io（JetStream）** 同时承担**竞拍核心**（出价命令的可靠处理）与**消息通知**（实时事件扇出），并**彻底移除 Redis**。目标读者是平台的竞拍参与者（出价方/卖家）与运维团队，价值在于：出价事件不丢失、可回放审计、高并发下可削峰扩展，并把中间件从「Redis + Postgres」简化为「NATS + Postgres」。

## Objectives

1. **可靠性**：出价与竞拍事件实现至少一次投递，节点重启/订阅者短暂离线**零事件丢失**（对比现状 Pub/Sub 丢失）。
2. **可扩展/削峰**：出价处理与通知解耦，支持多节点水平扩展；在峰值出价负载下 API 接受层不因落库阻塞而拒绝请求。
3. **可审计/可回溯**：出价事件持久化并可按竞拍回放，支撑历史重建与合规审计。
4. **简化技术栈**：移除 Redis，运维中间件种类 -1；出价核心与通知统一到 NATS。
5. **正确性守恒**：迁移后并发出价的金额单调性与竞态防护不弱于现状（Postgres 行锁 + 触发器保留）。

关键指标：事件丢失率、端到端出价→WS 通知延迟（P95）、出价峰值吞吐、重复出价率（应为 0）、DLQ 数量。

## User Stories

- 作为**出价方**，我希望提交出价后能可靠地在实时看板上看到自己与他人的最新出价，即使网络或服务短暂抖动也不漏更新。
- 作为**出价方**，我希望重复点击/重试提交不会造成重复出价或重复扣减，出价结果明确。
- 作为**卖家**，我希望竞拍开始/结束、最终成交等关键事件都能可靠推送给所有关注者。
- 作为**运维**，我希望在流量高峰时系统能削峰而非拒绝，且能查看出价事件积压与失败（DLQ）情况。
- 作为**合规/客服**，我希望能按某场竞拍回放全部出价事件以核对争议。

## Core Features

### 1. 可靠出价处理（竞拍核心）
出价请求进入持久化命令流，由后台处理器按竞拍有序、可靠地落库并产生领域事件。

- 1.1 提交出价后系统返回**已受理（202 Accepted）**，最终结果经 WebSocket 事件回推。
- 1.2 每次出价携带**幂等键**（客户端提供或服务端生成）；重复提交**不得**产生重复出价。
- 1.3 同一场竞拍的出价按提交顺序处理以保证公平性；不同竞拍可并行。
- 1.4 处理失败自动有限重试；仍失败的出价进入**死信（DLQ）**且不丢失，可事后排查。
- 1.5 并发出价的金额单调性与最终一致性不弱于现状。

### 2. 实时消息通知
竞拍领域事件可靠扇出到所有订阅该竞拍的 WebSocket 客户端。

- 2.1 支持事件类型：`BidPlaced`、`AuctionStarted`、`AuctionEnded`（对前端 WS 消息格式保持不变）。
- 2.2 通知具备持久化与重投能力，订阅者短暂离线后不漏关键事件。
- 2.3 客户端订阅/退订某场竞拍的行为与现状一致。

### 3. 事件持久化与回放
- 3.1 出价与竞拍事件持久保存一段可配置期限，支持按竞拍回放。
- 3.2 提供积压、重投、DLQ 等运行状态用于监控。

### 4. 移除 Redis
- 4.1 系统运行**不再依赖 Redis**；相关配置、依赖、容器编排与文档全部清理。
- 4.2 本地/CI 环境以 NATS 替代 Redis 一键启动。

## User Experience

- **出价方**：提交出价 → 即时得到「已受理」反馈 → 看板经 WebSocket 收到 `bid.placed` 更新（含自己与他人出价）。失败/被超越时同样经 WS 明确反馈。
- **卖家/观众**：进入竞拍页自动订阅，可靠接收开始/结束/成交事件。
- **交互一致性**：WebSocket 消息结构与订阅流程保持现状，前端改动集中在出价提交由「等待 201」改为「受理 202 + 监听 WS 结果」。
- **可访问性/包容性**：实时更新需提供非闪烁、可被屏幕阅读器识别的状态提示（如「出价已受理」「已被超越」），错误信息清晰可读、支持文本化。

## High-Level Technical Constraints

- **必用技术**：NATS.io（含 JetStream 持久化）作为竞拍核心与通知总线；不再使用 Redis。
- **现有系统对接**：Postgres 仍为事务权威（保留 `FOR UPDATE` + 触发器）；沿用 Go + fx + DDD 分层与 WebSocket 网关。
- **交付语义**：至少一次投递 + 幂等，业务效果等价精确一次。
- **性能/可扩展**：出价接受层非阻塞，可多节点水平扩展；端到端通知延迟需满足实时看板体验（P95 目标在 Tech Spec 细化）。
- **API 变更**：出价接口采用异步语义（202 + WS 回推）——已确认可接受为预期行为。
- **数据敏感性**：出价含用户与金额信息，事件流访问需鉴权，持久化事件保留期与清理策略需可配置。

## Out of Scope

- 出价 API 的向后兼容同步模式（本期仅异步 202；不提供 201 阻塞或双模式）。
- 更换底层数据库或改变出价事务的最终权威（仍为 Postgres）。
- 新增竞拍业务功能（代理出价、自动加价、拍卖类型扩展等）。
- 前端 UI 重构（仅适配出价提交的异步反馈）。
- 多区域/跨集群 NATS 超集群部署（后续考虑）。
- DLQ 的自动补偿/人工重放工具（本期仅保证不丢并可观测）。
